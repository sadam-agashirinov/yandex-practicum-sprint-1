Для перехода на микрофронтенд выбрал Webpack Module Federation.
Тип интеграции - Build time.
Тип композиции - клиентский.

Почему так:
1. Приложение содержит одну страницу на которой представлен весь функционал.
	Можно собрать сразу всю страницу и показать его клиенту. Фунционал не большой поэтому вес конечной сборки будет не большой.
2. Весь фронт использует один фреймворк - React. Использовать несколько JavaScript-фреймворков на одной странице - преимущество SPA. Нам это здесь не нужно.
3. Возможность использовать общий код во время выполнения.



Микрофронты.

0. host - Главный компонент объединяющий остальные Микрофронты.
1. mf-auth - микрофронтенд отвечающий за авторизацию и регистрацию.
2. mf-profile - микрофронтенд отвечающий за работу с профилем пользователя.
3. mf-card - микрофронтенд отвечающий за работу с карточками
4. shared - папка содержащая общий компоненты используемые Микрофронтами.


Задание №2 Разделение монолитного сервиса на микросервисы.

Файл с решением находится в корне и называется microservices-schema.drawio
Решение задачи продемострировано на вкладке "Решение задачи".
Вкладки находятся в нижней части окна браузера.

Общая идея следующая. Разделить монолит на микросервисы согласно DDD.
Каждый микросервис имеет набор функций для работы с доменом.
А так же имеет свою бд для хранения необходимой информации.
Возможно что некоторым сервисам понадобятся и NoSql базы. 
Но думаю их не принципиально указывать на данной схеме.

Работа с системой происходит через API шлюз (на схеме указан как API Gateway),
которая перенаправляет запросы в нужный микросервис.


Ниже буду некоторые размышления, 
которые я не смог указать схематически из-за не зная как это делать и на практике тоже не встречал.

1. На мой взгляд в системе должна быть шина сообщений (Kafka, RabbitMq) через которые проходять, например,
оповещения указанные в системе. Поэтому я их группировал в "Сервис оповещений".
Так же сюда наверное можно отнести и реализацию паттерна Saga, например, если не удалось купить товар\услугу чтоб сделать возврат денег
и восстановить значение кол-ва товаров\услуг.

2. Сервис товаров и Сервис услуг имеют функционал поиска.
Я думаю, что можно поиск в отдельный сервис вынести который хорошо под это заточен.
У него будет своя БД, ElasticSearch и мб другие механизмы ускоряющие поиск.
Так же добавить синхронизацию БД Сервиса товаров и услуг с БД Сервиса поиска, 
чтоб при выполнении CUD-операций (Create, Update, Delete) обновлялась инфа и в системе Сервиса оповещений.
